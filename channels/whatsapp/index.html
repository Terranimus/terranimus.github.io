WhatsApp

The odd one out — instead of a REST/WebSocket API, WhatsApp is driven through a sidecar process speaking the Baileys/WhatsApp Web protocol, paired to your account by scanning a QR code. No WhatsApp Business API account required.

Status & capabilities

Bidirectional, push-model, sidecar-process-backed. There's no bot token in the usual sense: WhatsApp Web pairing is per-device, so the "token" field in the credential vault is repurposed to carry a device label instead of a secret string — the actual pairing state (Baileys' multi-file auth) lives on disk under <data-dir>/whatsapp-auth/<label>, one directory per paired device. It's still routed through the same AES-256-GCM credential vault as every other channel, since losing that auth directory is equivalent to losing control of a bot token.

Auth setup

  1. Save a device label and default destination (a WhatsApp JID, not a phone number by itself) into the credential vault:
channel_save_credential(
  channel: "whatsapp",
  json_str: '{"token": "<device label, e.g. my-terransoul>", "chat_id": "<jid, e.g. 15551234567@s.whatsapp.net>"}'
)
  1. Flip channels.whatsapp.enabled to true. The sidecar supervisor starts on the next host tick.
  2. Watch the sidecar's log output for a QR code, and scan it from WhatsApp on your phone: Settings → Linked Devices → Link a Device. This is the same pairing flow WhatsApp Web itself uses.
  3. Once paired, the auth state persists on disk — you shouldn't need to re-scan unless you unlink the device from your phone or delete the auth directory.

Config

KeyMeaning
channels.whatsapp.enabledMust be true for the sidecar supervisor to start.

Troubleshooting

  • "channel not configured: pair a device first" — no device label saved yet, or the channel isn't enabled.
  • QR code never appears — check the sidecar process actually started (enable the channel, then watch the logs); it only prints a QR when it needs a fresh pairing.
  • Paired but sends fail — the destination JID format matters (<number>@s.whatsapp.net for a person, <id>@g.us for a group); a bare phone number will not resolve.
