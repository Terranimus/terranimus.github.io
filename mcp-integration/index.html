Using TerranSoul as an MCP server

TerranSoul isn't only an MCP client that reaches out to a model — it's also an MCP server. Any editor or agent that speaks Model Context Protocol can attach to its own brain and read/write the same memory the app itself uses.

Two transports, one brain

The server speaks the canonical MCP transport, newline-delimited JSON-RPC 2.0 over stdio — the default for Claude Desktop, the VS Code MCP extension, and Codex CLI. It also exposes an HTTP transport with bearer-token auth (loopback HTTP is reachable by any local process, so that one needs a token; stdio runs in a trusted parent/child relationship and doesn't).

Multiple agents can share one running brain rather than each spinning up their own. TerranSoul's own tooling reuses, in order: a release server on 127.0.0.1:7421, a tray server on :7423, or a dev server on :7422 — spawning terransoul --mcp-stdio as a subprocess bridges stdio messages to whichever of those is already live using its token file, instead of starting a second, disconnected brain.

npm run mcp                              # Tray on 127.0.0.1:7423
curl http://127.0.0.1:7423/health        # Verify
node scripts/mcp-tray-proxy.mjs --probe  # Show which server agents will use

Editor setup

TerranSoul can write its own MCP config into a target repo for VS Code (.vscode/mcp.json) or Zed (ACP-style context_servers config in .zed/settings.json) so you don't have to hand-author the JSON. It also exposes read-only MCP resources (e.g. a symbol's callers/callees) and guided prompts (find-references, impact-of-change, explain-cluster) on top of the tool list below — the same code-intelligence queries the app's own coding subsystem uses internally, reshaped for an editor to call directly.

What's callable

The tool list is capability-gated: without code-read access an MCP client sees 75 brain/knowledge-graph/observability tools; with full read/write + code-read it sees 94 — the same 75 plus 18 code-intelligence tools plus channel_send. These counts are pinned by tests in ai_integrations/mcp/tools.rs, not documentation that can silently drift.

The ones worth knowing by name:

ToolWhat it does
brain_searchHybrid + RRF (+ optional HyDE) search over memories. Tag/cognitive-kind filters narrow recall before any rerank. Rerank defaults to off — the raw RRF path is deterministic and fast (355–2325ms); the LLM-judge rerank is opt-in (rerank:true) and fail-open.
brain_suggest_contextThe flagship call: a curated context pack — top memories, knowledge-graph neighborhood, an LLM summary, and a delta-stable fingerprint for caching.
brain_execute_codeSandboxed script execution (Python/Node/shell/PowerShell) with bounded output. Gated behind the earned-autonomy action-trust ladder — deny-by-default until the category has earned trust. See Agentic coding.
brain_pursue_goalPush a standing objective onto the durable goal stack and step it forward across many turns until a confidence gate says done, or a budget runs out. See Goals & automation.
brain_delete_memoryDelete exactly one memory row by id — intentionally has no tag/pattern/bulk-selector field, so it can never touch more than the one id given.
Beyond those five, the surface covers knowledge-graph traversal (brain_kg_neighbors, brain_add_edge, brain_drilldown), skill learning (brain_optimize_skill, brain_get_skill), summarization, health checks, and the Canvas tools (canvas_navigate/canvas_eval/canvas_snapshot) that let an agent render generated content into a sandboxed panel. Call tools/list against a running server for the exhaustive, current schema.

Why this matters

Because retrieval happens over MCP instead of pasting whole files or whole conversation histories into context, an attached agent pulls only the slice of the brain relevant to what it's doing right now — context stays small as the corpus grows. That's also what lets TerranSoul, Claude Code, Copilot, Cursor, and Codex share one memory instead of each accumulating a separate, disconnected one.
