Goals & automation

"Pursue this across many turns until it's done" — an explicit primitive, not an implicit hope. Built entirely as orchestration over faculties that already existed; no new judge model, no new deliberation engine.

brain_pursue_goal

This is the hermes-agent /goal analog: push a goal, then the resident keeps stepping it forward, turn after turn, until a confidence gate says it's done or a budget runs out. Three existing pieces do the actual work:

  • The durable agenda (brain_goal_stack) — a prioritized stack of goals that survives across sessions, tracks progress, and can be suspended/resumed/closed.
  • The done/continue/abandon gate (brain_assess_confidence) — metacognitive confidence calibration: know when you don't know.
  • The compute allocator (brain_reason_budget) — sizes each continuation step's deliberation depth, so an easy step isn't over-thought and a hard one isn't under-thought.

brain_pursue_goal is what ties those three together into one callable loop:

opDoes
startPushes a new goal onto the stack (description required; priority/turn budget/wall-clock budget/due date all override brain-seeded defaults).
step (default)Re-enters the goal for one deliberation cycle. Syncs against the goal stack first — if something else suspended or closed the parent goal, this reports that honestly instead of fighting it. Enforces the turn/wall-clock budget second. When evidence is supplied, routes the done/continue/abandon decision through the confidence gate.
get / listRead pursuit state for one goal, or list every open pursuit.
self_tuneNudges the seeded default turn budget from logged outcomes.

Both the goal row and the pursuit row are ordinary durable SQLite rows — the loop survives an app restart for free, no separate persistence layer needed.

Commitments — the follow-ups nobody asked to be reminded about

A related but distinct faculty, brain_commitments, handles a different case: not a goal you explicitly asked to be pursued, but an inferred open-loop a conversation leaves dangling ("I'll let you know once the build finishes"). An LLM mentalizing pass supplies the inferred commitment; this faculty persists it and, on a periodic heartbeat, surfaces the due ones through the same attention/salience gate everything else in working memory competes through — governed by seeded quiet-hours and busy-skip policy so it doesn't nag.

Is there a cron-style scheduler too?

Yes, and it's worth being precise about which faculty is which. Underneath goal-pursuit there's a separate module (memory/automation.rs) that unifies a parameterized skill + a schedule into one typed "blueprint," and runs a consent-first suggestion queue: the brain can notice a routine you keep doing and propose scheduling it, but a suggestion only becomes a real scheduled_agents cron row when you explicitly accept it — nothing auto-schedules itself. This is real, shipped code with tests, but as of this writing it has no MCP tool or Tauri command exposing it directly the way brain_pursue_goal exposes goal-pursuit — it's currently an internal faculty the brain's own scheduling logic calls into, not something you invoke by name yet.

In short: brain_pursue_goal is the thing to reach for today if you want to hand the brain a standing objective and have it keep working the problem. The cron-suggestion queue is real but not yet a user-facing surface — check back here as that changes.
