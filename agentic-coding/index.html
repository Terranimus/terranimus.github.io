Agentic coding inside TerranSoul

TerranSoul carries a full agentic-coding subsystem under src-tauri/src/coding/ — roughly 70 modules covering sandboxed execution, checkpointed rollback, isolated worktrees, multi-agent workflows, and repo-aware search. brain_execute_code is the one MCP-visible entry point; most of the depth here sits below that single tool.

Sandboxed execution

brain_execute_code runs an agent-authored script (Python, Node, shell, or PowerShell) in an isolated child process, in its own process group, under a hard wall-clock timeout, and returns only a bounded, context-budget-safe tail of stdout — bulk intermediate output never re-enters the model's context window. It's a direct architectural analog of hermes-agent's execute_code tool, reimplemented in Rust against TerranSoul's own sandboxing primitives rather than vendored. Every execution is logged into the same procedural ledger the rest of the brain uses (content-hash deduped, success/failure counted), so a script that keeps succeeding accumulates evidence toward becoming a synthesized skill automatically. It's gated behind the earned-autonomy action-trust ladder — deny-by-default until that capability category has earned enough trust to run unsupervised.

Checkpoints, not just git

Before any file-mutating action, the engine commits the working tree into a shared, bare git object store that lives entirely outside your own .git — a browsable timeline of every turn, restorable independently of whatever you're doing in your real git history. Git's own content-addressable storage deduplicates blobs across projects and across turns, so this doesn't balloon disk usage. This sits alongside a separate in-run file snapshot mechanism that can roll back just the blocks touched by a single failed run.

Isolation when your checkout is dirty

If the working tree has uncommitted changes, the engine can spin up a temporary git worktree, run generated edits and tests there, and capture a patch for review — leaving your actual working tree untouched throughout.

Multi-agent workflows

Beyond single-script execution, there's a full multi-agent workflow system: specialized roles (Planner, Coder, Reviewer, Tester, Researcher, Orchestrator) collaborating through a YAML-serialized WorkflowPlan, each step naming its agent role, model, dependencies, and expected output shape. It follows the orchestrator-workers and evaluator-optimizer patterns (independent steps run in parallel; reviewer agents feed back into the plan), and every plan step is human-reviewable — you can edit, reassign the model, or approve/reject before execution proceeds. This is the same machinery the plugin marketplace's install/uninstall flow dispatches into.

Understanding a repo before touching it

A PageRank-based repo map builds a directed graph from indexed symbols and their call/import edges, ranks them by cross-reference importance, and renders a token-budgeted plain-text map with the most-referenced files and symbols first — so an agent (or the MCP terransoul://clusters resource) gets a repo overview sized to fit a context window instead of every file listed flat.

Everything else in the subsystem

The rest of coding/ covers the surrounding machinery: session replay, drift detection between what an agent claims and what actually changed, a code reviewer, symbol indexing and resolver/call-graph queries, diff/impact analysis, GitHub integration, a task queue, ambient (background, low-priority) task scheduling, cost tracking, and ledgered skill promotion. None of it is exposed as a single flashy feature — it's the load-bearing layer under brain_execute_code and the marketplace install flow.

How to reach it

As an end user, the entry points are brain_execute_code over MCP (see MCP integration) and whatever dispatches a WorkflowPlan — right now that's mostly internal (the marketplace install/uninstall flow). There's no standalone "run this workflow" UI yet; this page documents what exists in code today, not a roadmap.
